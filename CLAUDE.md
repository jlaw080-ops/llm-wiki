> 모든 작업 전 본 파일을 먼저 읽고, 본 규칙에 따라 수행합니다.

## 0. 핵심 철학

LLM Wiki의 본질은 **누적되는 영구 아티팩트**입니다. RAG처럼 매 쿼리마다 지식을 처음부터 재발견하지 않습니다.

- **사용자(Human)**: 자료 큐레이션, 탐색 방향 설정, 좋은 질문하기, 의미 해석
- **Claude(LLM)**: 요약, 상호 참조, 정리, 유지보수 — 모든 "잡일"
- **위키**: 매 자료와 매 질의로 점점 풍부해지는 영구 코드베이스

비유: **Obsidian = IDE, Claude = 프로그래머, 위키 = 코드베이스.**

## 1. 핵심 불변 규칙 (Invariants)

다음은 **어떤 경우에도 위반하지 않는 절대 규칙**이다. 모든 오퍼레이션의 전제 조건.

### INV-1. 작업 범위
- 모든 읽기/쓰기는 `90_LLMWiki/` 폴더 내부에서만 수행한다.
- vault의 다른 폴더(`00_DailyNotes/`, `01_Projects/` 등)는 절대 수정하지 않는다. 사용자가 명시적으로 요청한 경우에만 **읽기**만 가능.

### INV-2. raw/ 불변성
- `raw/` 폴더의 모든 파일은 **불변(immutable)**. 절대 수정/삭제하지 않는다.
- raw/ 안의 파일을 옮기거나 이름을 바꾸는 것도 금지.

### INV-3. INDEX.md 동기화 (페이지 생성/삭제/이동 시 필수)
- `wiki/` 내 페이지를 **생성**하면 즉시 INDEX.md에 항목 추가.
- 페이지를 **삭제**하면 즉시 INDEX.md에서 항목 제거.
- 페이지를 **이동**(카테고리 변경 포함)하면 즉시 INDEX.md 갱신.
- INDEX.md 갱신 없이 페이지 변경을 종료하지 않는다.

### INV-4. log.md 기록 (모든 오퍼레이션 필수)
- Ingest, Query, Lint **모든 오퍼레이션은 종료 직전 log.md에 항목 추가**.
- Query가 단순 회상이라도 기록한다 (위키가 어떻게 활용되는지 추적용).
- log.md 기록 없이 오퍼레이션을 종료하지 않는다.

### INV-5. 출처 의무
- 모든 사실 진술에는 출처가 있어야 한다. 출처 없는 주장은 작성 금지.
- 세부 규칙은 9장 참조.

### INV-6. 페이지 삭제 승인
- `wiki/` 내 페이지 삭제는 **사용자의 명시적 승인** 시에만 수행. 기본은 보존.

## 3. 폴더 구조

```
90_LLMWiki/
├── CLAUDE.md           ← 본 파일 (Schema)
├── INDEX.md            ← 위키 카탈로그 (모든 페이지 변경 시 동기 갱신)

├── raw/                ← 원본 자료 (불변)
│   ├── papers/
│   ├── standards/      ← 인증/기준 원문 (ZEB, PHI, KS 등)
│   ├── reports/        ← 사례 보고서, 시방서
│   ├── notes/          ← 회의록, 현장 메모
│   └── assets/         ← 이미지, 첨부파일
└── wiki/               ← LLM이 생성/유지하는 페이지
    ├── sources/        ← ⭐ 소스별 요약 (사실만, 해석 X)
    ├── concepts/       ← 개념·기술 (해석·종합 O)
    ├── standards/      ← 인증/기준 정리
    ├── equipment/      ← 설비/장비
    ├── projects/       ← 프로젝트 사례
    ├── people/         ← 연구자, 전문가
    ├── companies/      ← 회사, 기관
    ├── metrics/        ← 핵심 지표 (U-value, n50, 1차에너지소요량 등)
    └── synthesis/      ← 질의·탐색 결과 합성 페이지 (비교표, 분석 등)
```

## 4. ⭐ 소스 페이지와 개념 페이지의 분리 원칙

**가장 중요한 구조 원칙**. 이 분리가 위키의 신뢰도를 결정한다.

### 4.1 `wiki/sources/` — 소스 요약 페이지

- **하나의 raw 자료 = 하나의 sources 페이지** (1:1).
- **사실만 기록한다.** 해당 자료가 *무엇을 주장하는지*를 중립적으로 기술.
- 해석·종합·비판·LLM 일반 지식 보충 **금지**.
- 다른 소스나 개념 페이지를 인용하지 않는다 (단방향: raw → sources).
- 자료 내 수치·정의·주장을 그대로 기록.

**예**: `wiki/sources/passivhaus_principle_2023.md`는 *Passivhaus Institut, 2023* 자료가 무엇을 말하는지만 기록. PHIUS와의 차이나 한국 ZEB와의 비교 같은 건 절대 안 들어감.

### 4.2 `wiki/concepts/`, `standards/`, `equipment/` 등 — 개념 페이지

- **하나의 개념 = 하나의 페이지**, 여러 소스를 종합.
- 해석·종합·비교 가능.
- **모든 사실 진술은 `sources/` 페이지를 인용**해야 한다 (간접 인용 체인).
  - 인용 체인: `concepts/` → `sources/` → `raw/`
- 직접 raw/ 인용도 가능하지만, 가능하면 sources/를 경유한다.

### 4.3 `wiki/synthesis/` — 합성 페이지

- 사용자 질의나 탐색의 산출물 (비교표, 분석, Lint 보고서 등).
- 개념 페이지를 인용 (간접 인용 체인의 최상위).

### 4.4 인용 체인 시각화

```
raw/papers/passivhaus.pdf  (불변 원본)
         ↓ 인용
wiki/sources/passivhaus_2023.md  (사실만)
         ↓ 인용
wiki/concepts/패시브하우스.md  (해석·종합)
         ↓ 인용
wiki/synthesis/ZEB_vs_PHI.md  (탐색 결과)
```

본 위키는 다음 도메인을 다룬다:

## 6. 페이지 작성 규칙

### 6.1 프론트매터 (모든 페이지 필수, 예외 없음)

```yaml
---
title: 페이지 제목
type: llm-generated         # 항상 이 값 (사용자 직접 작성 노트와 구분)
category: concepts          # sources | concepts | standards | equipment | projects | people | companies | metrics | synthesis
tags: [패시브하우스, 기밀성능]
aliases: [n50, 기밀성능 측정, airtightness]
sources:                    # 인용한 sources 페이지 또는 raw 파일
  - wiki/sources/passivhaus_2023.md
  - raw/standards/PHI_criteria_2024.pdf
created: 2026-04-25
last_updated: 2026-04-25
confidence: high            # high | medium | low
status: stable              # draft | stable | needs-review
---
```

- `aliases`에는 한글/영문/약어를 모두 등록 → 위키링크가 유연히 동작.
- `confidence: low`는 출처가 빈약하거나 LLM 보충이 들어간 경우 표기.

### 6.2 페이지 본문 표준 템플릿 (개념 페이지)

```markdown
# {title}

> **요약**: 1~3문장. (Obsidian hover preview에 노출됨)

## 정의
명확한 정의. 가능한 한 원문 인용.

## 핵심 내용
- 핵심 포인트
- 가능한 한 [[다른 페이지]]로 상호 연결

## 관련 수치/기준
| 항목 | 값 | 기준/출처 |
|------|-----|----------|
| n50 | ≤ 0.6 /h | [[passivhaus_2023]] |

## 사례/적용
- [[프로젝트 페이지]]

## 관련 개념
- [[관련 페이지 1]]
- [[관련 페이지 2]]

## 출처
- [[passivhaus_2023]] — Passivhaus Institut 2023 보고서 요약
- [[PHI_criteria_2024]] — PHI 인증 기준 2024
```

### 6.3 소스 페이지 템플릿

```markdown
# {원자료 제목} ({연도})

> **출처**: [[raw/papers/passivhaus_principle.pdf]]
> **저자/발행**: Passivhaus Institut
> **유형**: 학술자료 | 인증기준 | 시방서 | 보고서 | 회의록

## 자료 개요
이 자료가 무엇에 관한 것인지 1~2문장.

## 주요 주장 / 핵심 내용
자료 내 진술을 사실 그대로 기록. 해석 금지.
- 자료는 ㅇㅇ을 ㅇㅇ로 정의한다 (p.12).
- 자료에 따르면 n50 기준은 0.6 /h 이하 (p.34).

## 핵심 수치/데이터
| 항목 | 값 | 페이지 |
|------|-----|------|
| n50 | ≤ 0.6 /h | p.34 |

## 인용된 페이지 (역참조용)
- 본 자료를 인용하는 개념 페이지 목록 (Claude가 자동 갱신)
- [[패시브하우스]]
- [[기밀성능]]
```

### 6.4 위키링크 규칙

- 새 개념 등장 시 **무조건 `[[ ]]`로 감싼다.** 페이지가 없어도 OK (broken link 추적 대상).
- 한국어/영어 동시 통용 시 **한국어를 메인 페이지명, 영어는 `aliases`에 등록**. 예: `[[열회수환기]]` (aliases: ERV, HRV).
- 모든 내부 참조는 wikilink 형식. 일반 마크다운 링크 `[text](url)`로 내부 페이지를 참조하지 않는다.

## 7. 운영 (Operations) — Ingest / Query / Lint

### 7.1 Ingest — 새 자료 통합

**트리거**: 사용자가 `raw/`에 파일 추가 후 "처리해줘" / "위키 업데이트" / "ingest" 지시.

**원칙**: 한 자료가 보통 **10~15개 페이지**에 영향을 미친다.

**워크플로우**:

1. **읽기**: `raw/`에서 신규/수정 파일 식별, 전체를 읽는다.
2. **핵심 논의**: 자료의 takeaway 3~5개를 사용자에게 보고하고 강조점 협의. 사용자가 "그냥 진행"이라 하면 다음 단계로.
3. **`sources/` 페이지 작성**: 자료당 한 페이지. **사실만**, 해석 없이.
4. **개념 페이지 생성/업데이트** (#9 원칙):
   - 동일 개념 페이지가 이미 존재? (제목 + 모든 alias 검색)
   - **존재 → 업데이트** (덮어쓰기 X, 보강 O).
   - **부재 → 신규 생성**.
   - **모든 사실 진술은 §7.1.3에서 만든 sources 페이지를 인용.**
5. **모순 처리** (8장 참조).
6. **상호 참조 갱신**: 새 페이지에서 기존 페이지로 가는 링크, 기존 페이지에서 새 페이지로 가는 링크 모두 점검.
7. **INDEX.md 갱신** (INV-3 의무).
8. **log.md 추가** (INV-4 의무).
9. **사용자 보고**: 신규/업데이트/검토필요 항목 요약.

### 7.2 Query — 위키에 질문하기

**트리거**: 사용자의 자연어 질문.

**검색 우선순위 (엄격)**:

1. **INDEX.md 먼저 읽기** — 관련 페이지 후보 식별. (이게 키 — 매 쿼리마다 모든 페이지를 스캔하지 않는다.)
2. 후보 **개념 페이지** 본문 읽기.
3. 더 깊이 필요하면 **소스 페이지(`sources/`)** 읽기.
4. **`raw/`는 마지막 수단** — 위키 내 정보가 부족하거나 신뢰성 검증이 필요한 경우에만 직접 접근.

**원칙**: raw/를 매번 직접 읽으면 LLM Wiki의 의미가 사라진다. 위키가 답을 못 주면 그건 **Lint 대상**(데이터 공백)이지, raw/ 직접 조회로 해결할 일이 아니다.

**답변 형식 옵션**:
- 일반 답변 → 채팅 응답 (markdown)
- 비교 요청 → 비교표
- 발표용 → Marp 슬라이드
- 데이터 시각화 → 차트

**⭐ 좋은 답변은 위키에 보존**:

쿼리 결과가 **새로운 합성·비교·연결**일 경우 사용자에게 제안:

> "이 비교표를 `wiki/synthesis/ZEB_vs_PHI_비교.md`로 저장할까요?"

승인 시 `synthesis/`에 페이지로 저장. **이 경우 INV-3에 따라 INDEX.md도 함께 갱신**.

**워크플로우 종료 단계**:
- (페이지 생성 시) INDEX.md 갱신 (INV-3)
- log.md 항목 추가 (INV-4) — Query 회상도 기록

### 7.3 Lint — 위키 건강성 점검

**트리거**: 사용자가 "점검해줘" / "lint" / "audit" 지시.

**점검 항목**:

1. **모순(Contradictions)**: 페이지 간 충돌하는 진술 → 보고만, 수정은 사용자 판단
2. **Stale 페이지**: 더 최신 자료에 의해 대체되었을 가능성
3. **고아 페이지(Orphan)**: 인바운드 링크가 없는 페이지
4. **Broken link**: `[[ ]]`로 감쌌으나 실제 페이지가 없는 항목
5. **누락된 상호 참조**: A 페이지에 B 개념이 언급되는데 `[[B]]` 링크가 없음
6. **데이터 공백**: 위키에 언급되었으나 충분히 다뤄지지 않은 주제 — 추가 자료 수집 또는 웹 검색으로 보완
7. **신뢰도 낮은 페이지**: `confidence: low` 페이지 목록
8. **소스-개념 분리 위반**: `sources/` 페이지에 해석이 들어갔거나, 개념 페이지에 출처 인용이 없는 경우
9. **추가 조사 제안**: "다음에 조사하면 좋을 주제" 3~5개

상세 보고서는 `wiki/synthesis/lint_YYYY-MM-DD.md`에 저장 (INV-3에 따라 INDEX.md 갱신, INV-4에 따라 log.md 기록).

**Lint는 진단이지 자동 수정이 아니다.** 모든 수정은 사용자 승인 후 진행.

## 8. 모순 처리 규칙

INV-5 + #6 요구사항.

### 8.1 모순 인지

Ingest 또는 Lint 중 자료 간 사실 충돌을 발견하면:
- 임의 판정 금지.
- 어느 쪽이 옳은지 LLM이 결정하지 않는다.

### 8.2 양쪽 소스 모두 인용 (필수)

해당 개념 페이지 하단에 다음 형식으로 기록:

```markdown
## ⚠️ 검토 필요 (모순)
```

## 9. 출처 추적 원칙

### 9.1 사실 vs 해석의 분리 (재강조)

- **`sources/` 페이지는 사실만.** 자료가 무엇을 주장하는지를 중립적으로 기록. 해석·비판·종합 금지.
- **개념 페이지는 해석·종합 가능**, 단 모든 사실 진술은 `sources/`나 `raw/` 인용 필요.

### 9.2 일반 원칙

1. **출처 의무**: 모든 사실 진술에 출처. 출처 없는 주장은 작성 금지 (INV-5).
2. **추측·일반 지식 보충 금지**: 원본에 없는 내용을 LLM 일반 지식으로 임의 보충 X. 보충이 불가피하면 본문에 `_(일반 상식 보충)_` 표기 + 프론트매터 `confidence: low`.
3. **수치는 원본 그대로.** 단위 변환 시 원 단위 병기. 예: "0.15 W/(m²·K) (원자료: U-value 0.15 W/m²K)".
4. **인용은 페이지 번호까지.** 가능한 한 `p.12` 식으로 명시.
5. **법규/기준 해석 금지.** 한국 건축 법령·고시·KS 등은 임의 해석 없이 원문 그대로 인용. 해석이 필요하면 `_(원문 직접 확인 필요)_` 표기.

## 10. INDEX.md 운영

### 10.1 역할
- 위키의 **카탈로그**. 쿼리 시 Claude가 **가장 먼저 읽는 파일**.
- INV-3에 따라 모든 페이지 생성/삭제/이동 시 동기 갱신.

### 10.2 항목 형식 (엄격)

**한 줄, 120자 이내, 다음 형식 준수**:

```
- [[페이지명]] — 1줄 요약 (sources: N)
```

- 줄바꿈 금지.
- "—"(em dash) 또는 "-"(hyphen)로 페이지명과 요약 구분.
- 요약은 페이지가 무엇인지를 한 호흡에 알 수 있게.
- 120자 초과 시 요약을 압축.

### 10.3 구조 예시

```markdown
# Wiki Index

마지막 갱신: 2026-04-25 | 페이지 수: 47

## Sources
- [[passivhaus_2023]] — Passivhaus Institut 2023 인증 기준 보고서 (raw: passivhaus_principle.pdf)
- [[korea_zeb_guide_2024]] — 한국 ZEB 인증 가이드라인 2024 (raw: zeb_kr_2024.pdf)

## Concepts
- [[열회수환기]] — 배기열로 급기를 예열/예냉하는 환기 방식 (sources: 3)
- [[기밀성능]] — 외피 공기 누설 정도, n50으로 측정 (sources: 2)

## Standards
- [[ZEB 인증]] — 한국 제로에너지건축물 인증 (1~5등급) (sources: 4)
- [[패시브하우스 PHI]] — Passivhaus Institut 국제 인증 (sources: 3)

## Equipment / Projects / People / Companies / Metrics / Synthesis
...
```

## 11. log.md 운영

### 11.1 원칙
- **단일 append-only 파일.** 기존 항목 수정·삭제 금지.
- INV-4에 따라 모든 오퍼레이션 종료 시 항목 추가.

### 11.2 항목 형식

```markdown
## [2026-04-25] ingest | passivhaus_principle.pdf
- 신규: [[passivhaus_2023]] (sources/), [[열교]], [[기밀성능]]
- 업데이트: [[패시브하우스]] (5대 원칙 보강)
- 모순 발견: [[기밀성능]] — n50 기준값 자료 간 상이
- INDEX 갱신: ✓

## [2026-04-26] query | "ZEB 1등급과 PHI 차이"
- 사용 페이지: [[ZEB 인증]], [[패시브하우스 PHI]]
- 답변 형식: 비교표
- 보존: [[ZEB_vs_PHI_비교]] (synthesis/)
- INDEX 갱신: ✓

## [2026-04-27] lint
- orphan 3건, broken link 5건, 모순 1건
- 상세: [[lint_2026-04-27]]
- INDEX 갱신: ✓
```

### 11.3 Prefix 규칙
`## [YYYY-MM-DD] {type} | {label}`
- `type`: `ingest` | `query` | `lint`
- 일관된 prefix를 유지해야 grep 등으로 파싱 가능 (예: `grep "^## \[" log.md | tail -10`).

- ❌ 오퍼레이션 종료 시 log.md 미기록 (INV-4)
- ❌ 기존 페이지 내용 통째로 덮어쓰기 (보강만 가능)

- ❌ 한국 건축 법규/기준의 임의 해석
- ❌ 모순 발견 시 임의 판단 또는 한쪽 소스만 인용
- ❌ Lint 결과 자동 수정
- ❌ log.md 기존 항목 수정 (append만 허용)
- ❌ INDEX.md 항목 120자 초과
- ❌ 내부 페이지를 일반 마크다운 링크 `[text](url)`로 참조 (wikilink만 허용)

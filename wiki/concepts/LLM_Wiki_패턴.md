## 3가지 핵심 오퍼레이션

### Ingest — 새 자료 통합

[[90_LLMWiki CLAUDE.md]] 기반. 한 자료가 보통 **10~15개 페이지**에 영향을 미친다.

**워크플로우 (9단계)**:

1. **읽기**: `raw/`에서 신규/수정 파일 전체를 읽음
2. **핵심 논의**: takeaway 3~5개를 사용자에게 보고, 강조점 협의
3. **`sources/` 페이지 작성**: 자료당 한 페이지. 사실만, 해석 없이
4. **개념 페이지 생성/업데이트**:
   - 동일 개념 페이지 존재 여부 확인 (제목 + alias 전수 검색)
   - 존재 → 보강(업데이트). 덮어쓰기 금지
   - 부재 → 신규 생성
   - 모든 사실 진술은 sources 페이지를 인용
5. **모순 처리**: 발견 시 임의 판정 금지, 양쪽 소스 인용 후 사용자에게 회부
6. **상호 참조 갱신**: 새↔기존 페이지 간 링크 양방향 점검
7. **INDEX.md 갱신** (INV-3)
8. **log.md 추가** (INV-4)
9. **사용자 보고**: 신규/업데이트/검토필요 항목 요약

### Query — 위키에 질문하기

**검색 우선순위 (엄격한 순서)**:

1. **INDEX.md 먼저** — 관련 페이지 후보 식별
2. **개념 페이지** 본문 읽기
3. **소스 페이지(`sources/`)** — 더 깊이 필요 시
4. **`raw/`는 마지막 수단** — 위키 내 정보가 부족하거나 신뢰성 검증 필요 시만

> raw/를 매번 직접 읽으면 LLM Wiki의 의미가 사라진다. 위키가 답을 못 주면 그건 **Lint 대상(데이터 공백)**이지, raw/ 직접 조회로 해결할 일이 아니다.

**⭐ 좋은 답변은 위키에 보존**: 새로운 합성·비교·연결이 나왔을 때 `wiki/synthesis/`에 저장 제안.

### Lint — 위키 건강성 점검

[[90_LLMWiki CLAUDE.md]] 기반. **Lint는 진단이지 자동 수정이 아니다.**

**9가지 점검 항목**:

1. **모순(Contradictions)**: 페이지 간 충돌하는 진술 → 보고만, 수정은 사용자 판단
2. **Stale 페이지**: 더 최신 자료에 의해 대체 가능성
3. **고아 페이지(Orphan)**: 인바운드 링크가 없는 페이지
4. **Broken link**: `[[ ]]`로 감쌌으나 실제 페이지가 없는 항목
5. **누락된 상호 참조**: 개념이 언급되었으나 링크가 없는 경우
6. **데이터 공백**: 언급되었으나 충분히 다뤄지지 않은 주제
7. **신뢰도 낮은 페이지**: `confidence: low` 페이지 목록
8. **소스-개념 분리 위반**: sources/에 해석, 또는 개념 페이지에 출처 인용 없는 경우
9. **추가 조사 제안**: 다음에 조사하면 좋을 주제 3~5개

## 소스-개념 페이지 분리 원칙

위키 신뢰도를 결정하는 핵심 구조 원칙. [[90_LLMWiki CLAUDE.md]] 기반:

| 페이지 종류 | 위치 | 기록 원칙 | 해석 허용 여부 |
|------------|------|-----------|--------------|
| 소스 요약 | `sources/` | 자료가 무엇을 주장하는지만 | ❌ 금지 |
| 개념 페이지 | `concepts/` | 여러 소스 종합 + 해석 | ✅ 허용 (출처 인용 필수) |
| 합성 페이지 | `synthesis/` | 질의·탐색 산출물 | ✅ 허용 |

## 인용 체인

출처 추적의 핵심 구조. [[90_LLMWiki CLAUDE.md]] 기반:

```
raw/ (불변 원본)
  ↓ 인용
wiki/sources/ (사실만)
  ↓ 인용
wiki/concepts/ (해석·종합)
  ↓ 인용
wiki/synthesis/ (탐색 결과)
```

- 모든 사실 진술은 체인을 역방향으로 추적할 수 있어야 한다
- concepts/에서 raw/를 직접 인용하는 것도 가능하지만 sources/를 경유하는 것이 원칙

## 모순 처리 패턴

[[90_LLMWiki CLAUDE.md]] 기반:

모순 발견 시 해당 개념 페이지 하단에 다음 형식으로 기록:

```markdown
## ⚠️ 검토 필요 (모순)
**쟁점**: (충돌 항목 설명)
- [[소스A]]: 주장A (p.N)
- [[소스B]]: 주장B (p.N)
**상태**: 미해결. 사용자 검토 필요.
**기록일**: YYYY-MM-DD
```

- 임의 판정 금지 — LLM이 어느 쪽이 옳은지 결정하지 않는다
- 양쪽 소스를 반드시 wikilink로 인용
- 사용자 결정 후 "## 과거 모순(해소됨)"으로 변환, 결정 근거 기록

## RAG와의 차이

| 구분 | RAG | LLM Wiki |
|------|-----|----------|
| 지식 저장 | 벡터 DB (비정형) | 마크다운 위키 (구조화) |
| 검색 방식 | 임베딩 유사도 | 위키링크 + 계층 탐색 |
| 축적 | 쿼리마다 재발견 | 매 Ingest마다 영구 축적 |
| 출처 추적 | 어려움 | raw → sources → concepts 체인 |
| 모순 처리 | 없음 | 명시적 모순 섹션 + 사용자 회부 |
| 건강성 점검 | 없음 | Lint 오퍼레이션 (9개 항목) |

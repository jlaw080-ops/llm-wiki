# LLM Wiki — Lint 스킬

## 핵심 철학

**Lint는 진단이지 자동 수정이 아니다.**

- 발견된 문제를 보고하고 우선순위를 제시한다.
- 모든 실제 수정(페이지 내용 변경, 링크 추가, 페이지 삭제)은 **사용자 명시적 승인 후** 진행.
- 자동 수정을 시도하면 INV-6(삭제 승인), INV-2(raw/ 불변성) 위반 위험이 생긴다.

유일한 자동 쓰기 작업: 보고서를 `wiki/synthesis/lint_YYYY-MM-DD.md`에 저장 + INDEX.md 갱신 + log.md 기록 (이 세 가지는 INV-3, INV-4 이행).

---

## 전제 조건

이 스킬을 실행하기 전에:

1. `CLAUDE.md` 읽기 (위키 구조·규칙 파악)
2. `INDEX.md` 읽기 (전체 페이지 목록 확보)
3. 현재 작업 디렉토리가 `{wiki_root}/` 인지 확인

---

## 핵심 불변 규칙 (Lint 중 준수)

| 규칙 | Lint에서의 적용 |
|------|----------------|
| INV-1 | 읽기/쓰기 모두 `{wiki_root}/` 내부에서만 |
| INV-2 | `raw/` 파일 수정 절대 금지 — 읽기만 가능 |
| INV-3 | 보고서(synthesis/) 저장 시 INDEX.md 즉시 갱신 |
| INV-4 | Lint 종료 시 log.md에 항목 추가 (필수) |
| INV-6 | 페이지 삭제 제안만 — 실행은 사용자 승인 후 |

---

## 실행 단계

### STEP 0 — 스캔 범위 확정

1. `INDEX.md`를 읽고 현재 등록된 모든 페이지 목록을 추출한다.
2. `wiki/` 하위 폴더를 Glob으로 스캔하여 **실제 존재하는 파일 목록**을 수집한다.
   - `wiki/sources/*.md`
   - `wiki/concepts/*.md`
   - `wiki/standards/*.md`
   - `wiki/equipment/*.md`
   - `wiki/projects/*.md`
   - `wiki/people/*.md`
   - `wiki/companies/*.md`
   - `wiki/metrics/*.md`
   - `wiki/synthesis/*.md`
3. INDEX 등록 목록 vs 실제 파일 목록을 비교해 두 가지 이상(anomaly)을 먼저 파악:
   - INDEX에 있으나 파일이 없는 항목 → STEP 2-1(Broken link) 선작업
   - 파일이 있으나 INDEX에 없는 항목 → "비공개 페이지" — 보고서에 기록

> 스캔 규모가 크면 카테고리별로 나눠서 읽는다.

---

### STEP 1 — 전체 페이지 내용 읽기

⚠️ **컨텍스트 절약 원칙 (필수 — 어기면 Autocompact thrashing 발생)**

1. **Grep 우선**: `[[링크]]`, `confidence:`, 해석성 표현 등 패턴은 Read 전에 Grep으로 먼저 추출한다.
2. **카테고리별 분할 읽기**: `sources/` → `concepts/` → `standards/` → 기타 순서. 한 카테고리 완료 후 결과 메모, 다음 카테고리 진행.
3. **50페이지 초과 시 배치 처리**: 카테고리 내 페이지가 많으면 10~15개씩 배치로 나눠 읽는다.
4. **점검 항목별 집중 읽기**: 한 번 읽을 때 하나의 수집 목표에만 집중한다.
   - `[[링크]]` 추출 → Grep `\[\[` 패턴으로 wiki/ 전체 스캔 후 집계
   - 프론트매터 점검 (`confidence`, `status`) → 각 파일 첫 20줄만 Read
   - 소스-개념 분리 위반 → sources/ 페이지에서 해석성 표현만 Grep
5. **Autocompact 재시작 후**: 미완료 카테고리부터 재개 (완료 카테고리 목록 파악 후).

INDEX에서 확인된 페이지를 카테고리별로 읽는다. Grep으로 선별 가능한 항목은 Read 전에 반드시 Grep을 먼저 실행한다.

**읽으면서 수집하는 데이터** (점검 항목별):

| 수집 항목 | 용도 |
|-----------|------|
| 페이지 내 `[[링크]]` 목록 | Broken link, 상호 참조 점검 |
| 프론트매터 `confidence`, `status` | 신뢰도 낮은 페이지 목록 |
| 프론트매터 `sources` 필드 | 소스-개념 분리 위반 점검 |
| 본문 내 해석·평가 표현 (sources/ 페이지에서) | 분리 원칙 위반 탐지 |
| 모든 사실 진술 패턴 (수치, 정의) | 모순·출처 누락 탐지 |
| 인바운드 링크 수 (다른 페이지에서 이 페이지를 가리키는 수) | 고아 페이지 탐지 |

---

### STEP 2 — 9개 항목 점검

각 항목을 독립적으로 점검하고, 발견 건을 내부적으로 메모한다.

---

#### 2-1. Broken Link (깨진 링크)

**정의**: `[[페이지명]]` 형식으로 참조되었으나 실제 `wiki/` 내에 해당 파일이 없는 항목.

**점검 방법**:
1. 모든 페이지에서 `[[...]]` 패턴을 추출.
2. 추출된 페이지명이 실제 파일로 존재하는지 확인.
   - `wiki/concepts/{name}.md`, `wiki/sources/{name}.md` 등 카테고리 전체 검색.
3. 없으면 → broken link 목록에 기록: `링크명 | 발견된 페이지 | 예상 카테고리`.

**보고 형식**:
```
### Broken Link (N건)
| 링크 | 발견 위치 | 예상 카테고리 |
|------|----------|-------------|
| [[열회수환기]] | wiki/concepts/ZEB.md | concepts/ |
```

**후속 제안**: "다음 Ingest 시 해당 개념 페이지 생성 대상으로 표시."

---

#### 2-2. 고아 페이지 (Orphan Pages)

**정의**: 다른 페이지에서 `[[이 페이지]]`로 참조하는 링크가 **하나도 없는** 페이지.

**점검 방법**:
1. 모든 페이지에서 `[[링크]]` 추출 → 링크가 가리키는 페이지명 집계.
2. wiki/ 내 각 페이지가 몇 번 참조되는지 카운트.
3. 인바운드 링크 = 0 인 페이지 → 고아 페이지 목록.

**예외**: `wiki/synthesis/` 페이지는 누적 탐색 결과물이므로 고아여도 정상. 보고는 하되 우선순위 낮음.

**보고 형식**:
```
### 고아 페이지 (N건)
| 페이지 | 카테고리 | 마지막 갱신 |
|--------|---------|-----------|
| [[GWAC_상호운용성모델]] | standards/ | 2026-04-25 |
```

---

#### 2-3. 모순 (Contradictions)

**정의**: 두 페이지 이상에서 동일 개념에 대해 **서로 다른 수치·정의·기준**을 주장하는 경우.

**점검 방법**:
1. 수치를 포함한 사실 진술 추출 (LLM 자연어 판단으로 추출: 숫자 + 단위 조합, 정의 문장 등).
2. 동일 개념 키워드가 여러 페이지에 등장할 때 값/표현을 비교.
3. 불일치 발견 시: 양쪽 페이지 + 값 + 출처 모두 기록.

**판단 기준**: 명백한 차이 (예: 한 페이지 n50 ≤ 0.6 /h, 다른 페이지 n50 ≤ 1.0 /h). 단순 서술 방식 차이는 모순이 아님.

**보고 형식**:
```
### 모순 (N건)
#### 모순 #1: [개념명]
- [[페이지A]]: "값 또는 진술 A" (출처: [[sourceX]])
- [[페이지B]]: "값 또는 진술 B" (출처: [[sourceY]])
- 권고: 사용자가 어느 출처를 우선할지 판단 후 → [[페이지A]] 또는 [[페이지B]]에 §8.2 형식으로 기록
```

**절대 금지**: 어느 쪽이 맞는지 LLM이 판정하지 않는다.

---

#### 2-4. Stale 페이지 (구식 가능성)

**정의**: `last_updated` 날짜가 오래되었거나, raw/ 에 더 최신 자료가 추가되었는데 해당 개념 페이지가 갱신되지 않은 경우.

**점검 방법**:
1. 모든 페이지의 `last_updated` 날짜를 수집.
2. log.md에서 가장 최근 ingest 날짜를 확인 (전체 Read 불필요 — `Grep "ingest |" log.md` 로 날짜만 추출).
3. ingest 날짜보다 `last_updated`가 오래된 페이지 중 해당 ingest에서 다뤄진 주제와 관련된 페이지 → stale 후보.
4. 정확한 판단은 불가능하므로 "업데이트 검토 필요" 수준으로만 보고.

**보고 형식**:
```
### Stale 페이지 검토 대상 (N건)
| 페이지 | last_updated | 관련 최근 ingest |
|--------|-------------|----------------|
| [[스마트그리드]] | 2026-04-25 | (최신) — 이상 없음 |
```

> 위키가 초기 단계라면 이 항목은 건너뛰거나 "해당 없음"으로 보고 가능.

---

#### 2-5. 누락된 상호 참조 (Missing Cross-references)

**정의**: 페이지 본문에 개념 A가 텍스트로 언급되지만 `[[A]]` 위키링크가 달려있지 않은 경우.

**점검 방법**:
1. INDEX.md에서 모든 페이지명과 aliases를 수집.
2. 각 페이지명에 대해 **Grep 두 번**을 실행한다:
   - `Grep "\[\[페이지명\]\]"` → 링크가 달린 등장 횟수 (wiki/ 전체 대상)
   - `Grep "페이지명"` → 총 등장 횟수 (wiki/ 전체 대상)
3. (총 등장 횟수) > (링크된 등장 횟수) → plain text 등장 존재 → 누락 상호 참조.
4. Grep 결과에서 발견된 파일과 문맥 줄을 보고 형식으로 기록.

> 페이지 수가 많을 때 각 페이지 본문을 Read로 전수 확인하지 않는다. Grep이 wiki/ 전체를 스캔하므로 Read보다 효율적이다.

**보고 형식**:
```
### 누락된 상호 참조 (N건)
| 발견 위치 | 누락 링크 | 등장 문맥 |
|----------|---------|---------|
| wiki/concepts/스마트그리드.md | [[SGAM]] | "...SGAM 아키텍처에서는..." |
```

---

#### 2-6. 데이터 공백 (Data Gaps)

**정의**: 위키 내에서 언급되었으나 전용 페이지가 없거나, 페이지가 있어도 `status: draft`이거나 내용이 지나치게 빈약한 주제.

**점검 방법**:
1. Broken link 목록 (2-1) → 페이지가 아예 없는 개념들.
2. `status: draft` 인 페이지 → 내용이 초안 단계.
3. 본문이 300자 미만인 개념 페이지 → 빈약한 페이지.
4. INDEX 항목 요약이 "<!-- 추가 필요 -->" 등 비어있는 카테고리.

**보고 형식**:
```
### 데이터 공백 (N건)
| 주제 | 공백 유형 | 권고 조치 |
|------|---------|---------|
| [[열회수환기]] | 페이지 없음 (broken link) | raw/ 자료 수집 후 /ingest |
| [[ZEB_인증]] | draft 상태, 내용 미비 | 관련 기준서 수집 권장 |
```

---

#### 2-7. 신뢰도 낮은 페이지 (Low Confidence)

**정의**: 프론트매터에 `confidence: low`로 표시된 페이지.

**점검 방법**:
1. 모든 페이지 프론트매터에서 `confidence: low` 항목 수집.
2. 해당 페이지의 `sources` 필드 확인 — 인용 소스가 없거나 `_(일반 상식 보충)_` 표기가 있는지 확인.

**보고 형식**:
```
### 신뢰도 낮은 페이지 (N건)
| 페이지 | 이유 | 권고 |
|--------|------|------|
| [[Andrej_Karpathy]] | 원자료 캡처 부분적, LLM 보충 포함 | 원문 재수집 후 보강 |
```

---

#### 2-8. 소스-개념 분리 원칙 위반

**정의** (CLAUDE.md §4):
- `wiki/sources/` 페이지에 **해석·종합·비교** 문장이 포함된 경우.
- `wiki/concepts/` 등 개념 페이지에 `[[sources_페이지]]` 인용 **없는 사실 진술**이 있는 경우.

**점검 방법**:

*sources/ 위반 탐지*:
- "따라서", "즉", "결론적으로", "비교하면", "반면", "더 나은" 등 해석성 표현을 sources/ 페이지 본문에서 탐지.

*concepts/ 위반 탐지*:
- 수치, 정의, 기준값이 포함된 문장에 `[[...]]` 인용이 없는 경우.
- 프론트매터 `sources` 필드가 비어있는 개념 페이지.

**보고 형식**:
```
### 소스-개념 분리 위반 (N건)
| 페이지 | 위반 유형 | 문제 문장 (발췌) |
|--------|---------|---------------|
| wiki/sources/karpathy_llm_wiki_gist.md | sources/ 내 해석 | "따라서 LLM Wiki가 RAG보다 우수하다" |
| wiki/concepts/상호운용성.md | 출처 없는 수치 | "상호운용성 레벨은 6단계로 구분된다" |
```

---

#### 2-9. 추가 조사 제안

**정의**: 현재 위키 내용을 바탕으로 "다음에 수집하면 유용할 주제" 3~5개를 제안한다.

**판단 기준**:
- 현재 위키에서 자주 참조되는데 내용이 빈약한 주제.
- 사용자 도메인(신재생에너지, 건물에너지, ZEB, 패시브하우스)과 연관성 높은 공백.
- log.md Query 항목에서 반복 등장하는 주제.

**보고 형식**:
```
### 추가 조사 제안 (3~5개)
1. **[주제명]** — 현재 위키에서 N회 참조되지만 전용 페이지 없음. 권장 출처: [예시]
2. **[주제명]** — 관련 개념 페이지는 있으나 기준 수치 미비.
...
```

1. 🔴 **높음**: {Broken link N건, 모순 N건, 소스-개념 분리 위반 N건} — 위키 신뢰도에 직접 영향

3. 🟢 **낮음**: {Stale N건, 신뢰도 낮은 페이지 N건} — 순차 개선

> 이 보고서의 모든 항목은 **진단 결과**입니다.  
> 수정이 필요한 항목이 있으면 사용자 지시에 따라 별도 진행합니다.
```

  - 고아 페이지: N건
  - 누락 상호 참조: N건
  - 데이터 공백: N건

  - Stale 가능성: N건
  - 신뢰도 낮은 페이지: N건

```

## 수정 작업 가이드 (사용자 승인 후)

| 항목 | 수정 방법 | 주의사항 |
|------|---------|---------|
| Broken link | 해당 개념 페이지 신규 생성 → /ingest 트리거 권장 | raw/ 없으면 `confidence: low`로 시작 |
| 고아 페이지 | 관련 개념 페이지에서 `[[이 페이지]]` 링크 추가 | 의미 있는 컨텍스트에서만 연결 |
| 모순 | CLAUDE.md §8.2 형식으로 `## ⚠️ 검토 필요` 섹션 추가 | 어느 쪽이 맞는지 LLM이 판정 금지 |
| 누락 상호 참조 | plain text → `[[위키링크]]`로 교체 | 의미 없는 단순 언급은 링크 안 달아도 됨 |
| 소스-개념 분리 위반 | sources/ 내 해석 문장 → 사실 진술로 교체 / 개념 페이지 출처 추가 | sources/ 내용 삭제보다 교체 우선 |
| 데이터 공백 | 관련 자료 raw/에 추가 후 /ingest 실행 | |
| 신뢰도 낮은 페이지 | 출처 보강 후 `confidence: medium/high`로 업데이트 | |

## 금지 사항

- ❌ Lint 결과를 사용자 승인 없이 자동 수정
- ❌ `raw/` 파일 수정 (INV-2)
- ❌ 모순 발견 시 임의 판정 후 한쪽 삭제
- ❌ 고아 페이지 자동 삭제 (INV-6)
- ❌ 보고서 저장 후 INDEX.md 미갱신 (INV-3)
- ❌ log.md 미기록으로 Lint 종료 (INV-4)
- ❌ `sources/` 내 위반 문장을 임의로 삭제 (교체만 가능하며, 반드시 승인 후)

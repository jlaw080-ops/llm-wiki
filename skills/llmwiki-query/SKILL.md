# LLM Wiki — Query 스킬

## 핵심 철학

LLM Wiki의 Query는 RAG가 아니다.

- **RAG**: 매 질문마다 벡터 유사도로 청크를 검색 → 매번 처음부터 재발견
- **LLM Wiki Query**: 누적된 위키 페이지를 직접 읽고 탐색 → 위키가 쌓일수록 답변 품질 향상

위키가 답을 못 주면 그건 **Lint 대상(데이터 공백)**이지, raw/ 직접 조회로 해결할 일이 아니다.

---

## 전제 조건

이 스킬을 실행하기 전에:

1. `CLAUDE.md` 를 읽고 위키 도메인·폴더 구조 파악 (세션 시작 시 1회)
2. 현재 작업 디렉토리가 `{wiki_root}/` 인지 확인

---

## 핵심 불변 규칙 (Query 중 준수)

| 규칙 | Query에서의 적용 |
|------|----------------|
| INV-1 | 읽기/쓰기 모두 `{wiki_root}/` 내부에서만 |
| INV-2 | `raw/` 는 마지막 수단. 일상적 Query에서 직접 읽지 않는다 |
| INV-3 | synthesis/ 페이지 저장 시 INDEX.md 즉시 갱신 |
| INV-4 | Query 종료 시 log.md에 항목 추가 (단순 회상도 기록) |
| INV-5 | 답변 내 모든 사실 진술에 출처 위키링크 표기 |

---

## 검색 우선순위 (엄격한 순서)

```
1순위: INDEX.md        → 관련 페이지 후보 식별
2순위: 개념 페이지      → wiki/concepts/, standards/, equipment/ 등
3순위: 소스 페이지      → wiki/sources/ (더 깊은 사실 확인 필요 시)
4순위: raw/            → 위키 정보 부족 or 신뢰성 검증 필요 시만 (최후 수단)
```

**raw/ 를 먼저 읽는 것은 LLM Wiki 패턴 위반이다.**  
raw/ 에 답이 있어도 위키 페이지가 없으면 Lint 보고서에 "데이터 공백"으로 기록해야 한다.

---

## 실행 단계

### STEP 1 — 질문 분석

사용자 질문에서 다음을 추출한다:

- **핵심 개념**: 질문에서 검색할 키워드/개념명 (한국어·영어·약어 모두)
- **질문 유형**: 정의 | 비교 | 수치/기준 | 사례 | 관계 | 종합/분석
- **예상 출력 형식**: 텍스트 답변 | 표 | 목록 | 슬라이드 | 코드

---

### STEP 2 — INDEX.md 탐색

1. `INDEX.md` 를 읽어 후보 페이지를 식별한다.
   - **위키가 100페이지 이상이거나 INDEX.md가 200줄 초과인 경우**: 전체 Read 대신 `Grep "개념명" INDEX.md` 로 관련 항목만 추출한다.
2. STEP 1에서 추출한 개념명과 INDEX 항목을 매칭:
   - 직접 매칭 (페이지명 일치)
   - 별칭 매칭 (aliases 포함 여부)
   - 관련 카테고리 (standards/, concepts/ 등) 확인
3. 후보 페이지 목록을 내부적으로 메모.

> INDEX에 관련 항목이 전혀 없으면 → STEP 5(데이터 공백 처리)로 이동.

---

### STEP 3 — 개념 페이지 읽기

INDEX에서 식별된 후보 페이지를 읽는다.

**읽기 우선순위**:
1. 질문과 직접 관련된 개념 페이지
2. 해당 페이지의 `## 관련 개념` 섹션 → 필요 시 연결 페이지도 읽기
3. `## 출처` 섹션 → 더 깊은 사실 확인이 필요하면 STEP 4로

**읽으면서 수집**:
- 관련 사실·정의·수치
- 해당 사실의 출처 wikilink (`[[sources_페이지명]]`)
- 페이지 내 모순 표시 (`## ⚠️ 검토 필요`) 여부

---

### STEP 4 — 소스 페이지 읽기 (필요 시)

다음 경우에만 `wiki/sources/` 페이지를 추가로 읽는다:
- 개념 페이지의 사실이 불충분하거나 수치가 불명확한 경우
- 사용자가 "출처 확인", "원문 확인"을 요청한 경우
- 답변에서 특정 수치·정의의 정확한 출처 페이지 번호가 필요한 경우

`raw/` 직접 읽기는 **소스 페이지에도 정보가 없거나** 신뢰성 검증이 명시적으로 요청된 경우에만.

---

### STEP 5 — 답변 생성

**답변 형식 선택** (질문 유형별):

| 질문 유형 | 형식 |
|---------|------|
| 정의/개념 | 마크다운 텍스트, 관련 링크 포함 |
| 비교 | 비교표 (항목 vs 항목) |
| 수치/기준 | 표 (항목 / 값 / 출처) |
| 사례/프로젝트 | 목록 or 카드형 설명 |
| 종합/분석 | 구조화된 본문 + 소결론 |
| 발표용 | Marp 슬라이드 제안 |

**답변 본문 규칙**:
- 위키에서 확인된 사실 → `[[페이지명]]` 으로 출처 표기
- 위키 외 일반 지식 보충 → `_(위키 외 일반 상식)_` 표기 + 신뢰도 주의 안내
- 모순이 발견된 개념 → "⚠️ 이 항목은 위키 내 검토 필요 상태입니다: [[페이지명]]" 안내

**데이터 공백 처리** (위키에 관련 내용 없을 때):

```
위키에 [{질문 핵심 개념}] 관련 페이지가 없습니다.

알고 있는 일반 정보로 답변하되, 위키에 이 주제를 추가할 것을 권장합니다.
→ raw/에 관련 자료를 추가하고 /ingest 를 실행하면 위키에 영구 축적됩니다.

[일반 정보 기반 답변]
...
```

---

### STEP 6 — 합성 결과 보존 제안 (⭐ 핵심)

쿼리 결과가 다음 중 하나에 해당하면 보존을 제안한다:

- 새로운 **비교·대조** (두 개 이상 개념 비교표)
- 새로운 **연결·관계** 발견 (A와 B의 관계가 위키에 명시되지 않은 경우)
- 복잡한 **종합 분석** (여러 페이지를 읽어야 얻어지는 결론)
- 반복 활용 가능한 **참고표·기준 정리**

제안 형식:
```
이 답변을 wiki/synthesis/{제안_파일명}.md 로 저장할까요?
다음에 같은 주제를 물을 때 바로 참조할 수 있습니다.
```

사용자 승인 시 → 아래 synthesis 페이지 템플릿으로 저장.

**synthesis 페이지 템플릿**:

```markdown
---
title: {합성 제목}
type: llm-generated
category: synthesis
tags: [{관련 태그}]
sources:
  - wiki/concepts/{관련 페이지}.md
created: {YYYY-MM-DD}
last_updated: {YYYY-MM-DD}
query: "{원래 질문}"
confidence: medium
status: stable
---

# {합성 제목}

> **생성 경위**: {원래 질문} 에 대한 Query 결과 합성 ({날짜})

{답변 본문}

## 출처
- [[{개념 페이지 1}]]
- [[{sources 페이지}]]
```

synthesis 저장 시 → **INV-3에 따라 INDEX.md 즉시 갱신** (Synthesis 섹션에 항목 추가).

---

### STEP 7 — log.md 기록 (INV-4 필수)

Query가 단순 회상이라도 반드시 기록한다.

```markdown
## [{YYYY-MM-DD}] query | "{질문 요약}"
- 사용 페이지: [[{페이지1}]], [[{페이지2}]]
- 답변 형식: {텍스트|비교표|목록 등}
- 보존: {없음 | [[{synthesis 페이지명}]] (synthesis/)}
- 데이터 공백: {없음 | {주제} — 위키에 미수록}
- INDEX 갱신: {해당 없음 | ✓}
```

---

## 답변 품질 기준

| 기준 | 좋은 답변 | 나쁜 답변 |
|------|---------|---------|
| 출처 | 모든 사실에 `[[위키링크]]` | "일반적으로 ..." (출처 없음) |
| 범위 | 위키 내 내용을 최대한 활용 | raw/ 직접 읽기로 우회 |
| 한계 인정 | 위키에 없으면 "데이터 공백" 명시 | 있는 척 일반 지식으로 채움 |
| 모순 | ⚠️ 모순 상태 그대로 전달 | 한쪽 판정해서 답변 |
| 재활용 | 좋은 합성 결과를 synthesis/에 저장 제안 | 매번 새로 계산 |

---

## 자주 있는 패턴

### 비교 질문 ("A와 B의 차이는?")

1. INDEX에서 A, B 각각 검색
2. 두 개념 페이지를 읽고 비교 축 식별
3. 비교표 작성 — 각 셀에 출처 wikilink
4. synthesis/ 저장 제안

### 수치 기준 질문 ("n50 기준이 얼마야?")

1. INDEX에서 기준/수치 관련 페이지 검색
2. 개념 페이지의 수치 확인 → sources 페이지에서 원문 검증
3. 출처 명시 (소스 + 페이지 번호)
4. 모순이 있으면 ⚠️ 표시

### 개념 설명 질문 ("ZEB가 뭐야?")

1. INDEX → 해당 개념 페이지
2. 정의 + 핵심 내용 + 관련 개념 연결
3. 필요 시 standards/ 페이지도 참조

### 위키에 없는 질문

1. "위키에 관련 내용이 없습니다" 명시
2. 일반 지식 기반 간략 답변 (_(위키 외)_ 표기)
3. `/ingest` 로 보완 권장

---

## 금지 사항

- ❌ `raw/` 를 기본 검색 경로로 사용
- ❌ 위키 없이 LLM 일반 지식만으로 사실 진술 (출처 표기 없이)
- ❌ 모순 상태 페이지에서 임의로 한쪽 선택해 답변
- ❌ synthesis 저장 시 INDEX.md 미갱신
- ❌ log.md 미기록으로 Query 종료
- ❌ `raw/` 파일 수정 (Query 중 읽기도 최후 수단)

# LLM Wiki — Ingest 스킬

## 전제 조건

이 스킬은 다음 구조를 가진 LLM Wiki 폴더에서 동작한다:

```
{wiki_root}/
├── CLAUDE.md     ← LLM Wiki 운영 규칙
├── INDEX.md      ← 위키 카탈로그
├── log.md        ← append-only 변경 이력
├── raw/          ← 원본 자료 (불변)
└── wiki/
    ├── sources/
    ├── concepts/
    ├── standards/
    ├── equipment/
    ├── projects/
    ├── people/
    ├── companies/
    ├── metrics/
    └── synthesis/
```

**이 스킬을 실행하기 전에 반드시 확인:**
- 현재 작업 디렉토리가 `{wiki_root}/` 인지 확인
- `CLAUDE.md` 를 읽고 해당 위키의 INV 규칙·도메인·폴더 구조를 파악
- `INDEX.md` 를 읽고 현재 위키 상태(페이지 수, 기존 페이지 목록) 파악
- `log.md` 의 최근 10~20개 항목을 읽어 이미 인제스트된 파일 목록 파악

---

## ⚠️ 컨텍스트 절약 원칙 (모든 단계 최우선 준수)

**이 규칙을 어기면 Autocompact thrashing이 발생한다.**

1. **한 번에 한 파일만 읽는다.** 여러 wiki 파일을 동시에 Read하지 않는다.
2. **파일 원문을 컨텍스트에 누적하지 않는다.** 읽은 즉시 핵심만 메모하고, 다음 도구 호출 전에 원문은 폐기한다.
3. **Grep이 Read보다 우선이다.** 페이지 전체가 필요하지 않으면 Grep으로 필요한 줄만 찾는다.
4. **소스 페이지는 100줄 이하로 작성한다.** 길게 쓰면 다음 세션에 읽을 수 없다.
5. **Autocompact 재시작 후 복구 절차 (필수)**:
   - `log.md` 마지막 30줄만 읽어 마지막 완료 파일 확인
   - 다음 처리 대상 파일 1개만 특정
   - 관련 concept 페이지는 필요한 시점에 **섹션 단위로만** 읽는다
   - 재시작 직후 여러 위키 파일을 한꺼번에 Read하는 것 금지

---

## 핵심 불변 규칙 (모든 단계에서 준수)

| 규칙 | 내용 |
|------|------|
| INV-1 | 모든 읽기/쓰기는 `{wiki_root}/` 내부에서만 |
| INV-2 | `raw/` 파일 수정·삭제·이동·이름변경 절대 금지 |
| INV-3 | 페이지 생성/삭제/이동 시 `INDEX.md` 즉시 갱신 |
| INV-4 | 모든 오퍼레이션 종료 시 `log.md` 에 항목 추가 |
| INV-5 | 모든 사실 진술에 출처 필수. 출처 없는 주장 금지 |
| INV-6 | `wiki/` 내 페이지 삭제는 사용자 명시적 승인 시에만 |

## 실행 단계

### STEP 0 — 미처리 파일 감지

1. `raw/` 폴더 전체를 재귀적으로 스캔한다.
2. `log.md`에서 인제스트된 파일명을 추출한다.
   - **Grep `"## \["` 패턴으로 log.md의 모든 인제스트 헤더를 추출** → 파일명 파싱.
   - 특정 파일명 확인이 필요하면 `Grep "파일명" log.md` 로 직접 검색.
   - **log.md 전체를 Read하지 않는다** (컨텍스트 낭비).
3. **미처리 파일 = raw/에 있으나 log.md에 없는 파일.**
4. 미처리 파일 목록을 사용자에게 보여주고 처리 순서를 확인한다.

> 미처리 파일이 없으면: "현재 raw/에 미처리 파일이 없습니다." 보고 후 종료. log.md에 점검 항목 추가.

---

### STEP 1 — 파일 읽기 (INV-2 준수)

- 미처리 파일을 **읽기 전용**으로 읽는다. **절대 수정하지 않는다.**
- 파일이 여럿이면 한 번에 하나씩 처리한다 (배치 처리 금지 — 각 파일마다 사용자 Q&A 필요).
- 파일 형식별 처리:

  **`.md` / `.txt`**
  - Read 도구로 직접 읽기
  - 파일이 500줄 초과인 경우: 처음 200줄 읽기 → 요약 메모 → 다음 200줄 읽기 → 반복
  - 읽은 청크는 즉시 요약하고 원문은 컨텍스트에서 해제 (전체 원문을 메모리에 유지하지 않음)

  **`.pdf`**
  1. 먼저 `mcp__pdf-mcp__pdf_info` 로 전체 페이지 수 확인
  2. **20페이지 이하**: `mcp__pdf-mcp__pdf_read_all` 사용
  3. **21페이지 이상**: `mcp__pdf-mcp__pdf_read_pages` 로 **15페이지 단위** 청크 읽기
     - 청크 1 읽기 → 핵심 내용 메모 → 청크 2 읽기 → 핵심 내용 메모 → 반복
     - 각 청크의 원문은 요약 후 다음 청크 읽기 전에 폐기
     - 모든 청크 완료 후 메모들을 합산해 STEP 2 입력으로 사용
  4. **100페이지 이상**: 목차(`mcp__pdf-mcp__pdf_get_toc`) 먼저 확인 → 핵심 섹션만 선택적으로 읽기 → 사용자에게 읽은 범위 고지

  **이미지 / 바이너리**
  - 내용 파악 불가 사실을 사용자에게 알린 후 건너뜀

> ⚠️ **컨텍스트 과부하 방지 원칙**: 파일 원문 전체를 컨텍스트에 누적하지 않는다.
> 청크를 읽을 때마다 핵심 내용만 메모로 남기고, 다음 청크 읽기 전에 이전 원문은 버린다.

---

### STEP 2 — 핵심 논의 (사용자에게 보고)

읽은 파일의 내용을 분석하여 다음을 사용자에게 제시한다:

**A. 소스 요약 (3~5개 항목)**

```
📄 파일명: {파일명}
📎 추정 출처/저자: {추정값}
📌 핵심 주장 (Key Claims):
  1. ...
  2. ...
  3. ...
🏷️ 언급된 엔티티 (사람/도구/조직): ...
💡 다루는 개념 (Concepts): ...
```

**B. 사용자 Q&A (3가지 질문)**

다음 세 가지를 반드시 질문한다:

```
Q1. 이 글/자료를 왜 캡처하셨나요?
    (어떤 업무/프로젝트 맥락에서 필요한 자료인가요?)

Q2. 현재 진행 중인 작업이나 관심사와 어떻게 연결되나요?

Q3. 위키에 특히 강조해서 담고 싶은 부분이 있나요?
    (없으면 "전반적으로 처리해줘"라고 말씀해주세요.)
```

> 사용자가 "그냥 진행" / "전반적으로 처리" / "바로 해줘" 라고 하면 Q&A 생략하고 STEP 3으로 진행.

---

### STEP 3 — wiki/ 반영

#### 3-1. `wiki/sources/` 페이지 작성

**규칙**: 사실만, 해석 금지. 자료가 무엇을 주장하는지만 중립적으로 기록.

```markdown
---
title: {자료 제목} ({연도 또는 날짜})
type: llm-generated
category: sources
tags: [{관련 태그}]
aliases: [{대체 검색어}]
sources:
  - raw/{경로}/{파일명}
created: {오늘 날짜 YYYY-MM-DD}
last_updated: {오늘 날짜}
confidence: {high|medium|low}
status: {stable|draft|needs-review}
---

# {자료 제목}

> **출처**: [[raw/{경로}/{파일명}]]
> **원본 URL**: {URL — 웹 클립인 경우}
> **저자/발행**: {저자}
> **유형**: {학술자료|인증기준|시방서|보고서|회의록|웹아티클}

## 자료 개요
{이 자료가 무엇에 관한 것인지 1~2문장}

## 주요 주장 / 핵심 내용
- 자료는 ㅇㅇ을 ㅇㅇ로 정의한다 (p.N).
- 자료에 따르면 ... (p.N).

## 핵심 수치/데이터
| 항목 | 값 | 페이지/출처 |
|------|-----|------------|
| ... | ... | p.N |

## ⚠️ 데이터 공백 (해당 시)
{원문 누락, 캡처 불완전 등 한계 기록}

## 인용된 페이지 (역참조용)
{이 sources 페이지를 인용하는 concept 페이지 — 추후 Claude가 갱신}
```

**소스 페이지 저장 경로**: `wiki/sources/{snake_case_파일명}.md`

> ⚠️ **소스 페이지 길이 상한: 100줄 (프론트매터 포함)**
> - 주요 주장은 최대 8개 bullet로 제한
> - 핵심 수치/데이터 표는 최대 10행으로 제한
> - 100줄 초과 시 내용을 압축하여 핵심만 남긴다 (전문 포함 금지)
> - 출처가 불명확하거나 캡처가 불완전한 경우: `confidence: low`, `status: needs-review` 설정
> - LLM 일반 지식으로 보충한 경우: 해당 내용 뒤에 `_(일반 상식 보충)_` 표기

---

#### 3-2. 개념 페이지 생성 / 업데이트

자료에서 식별된 각 핵심 개념에 대해:

**A. 기존 페이지 존재 여부 확인 (필수)**

1. `INDEX.md` 에서 개념명 검색 — **전체 읽기 금지, Grep으로 해당 줄만 검색**
2. `wiki/concepts/`, `wiki/standards/` 등에서 파일명 + aliases Grep 검색
3. 한국어·영어·약어 변형으로도 Grep

**B. 존재하면 → 업데이트 (덮어쓰기 금지, 보강만)**
- 기존 페이지를 Read할 때 **전체 파일을 한 번에 읽지 않는다**
  - 먼저 프론트매터(1~20줄)만 읽어 sources 목록 파악
  - 추가·수정이 필요한 섹션만 Grep으로 위치 확인 후 해당 줄 범위만 Read
  - Edit 도구로 해당 섹션만 수정 (전체 파일 재작성 금지)
- `sources` 프론트매터에 새 sources 페이지 추가
- `last_updated` 갱신

**C. 없으면 → 신규 생성**

```markdown
---
title: {개념명}
type: llm-generated
category: {concepts|standards|equipment|people|companies|metrics}
tags: [{관련 태그}]
aliases: [{한국어, 영어, 약어 모두}]
sources:
  - wiki/sources/{sources_페이지명}.md
created: {오늘 날짜}
last_updated: {오늘 날짜}
confidence: {high|medium|low}
status: {stable|draft}
---

# {개념명}

> **요약**: 1~3문장. (Obsidian hover preview에 노출)

## 정의
{명확한 정의. 가능한 한 원문 인용.} [[sources_페이지명]]

## 핵심 내용
- ...  [[sources_페이지명]]

## 관련 수치/기준
| 항목 | 값 | 기준/출처 |
|------|-----|----------|
| ... | ... | [[sources_페이지명]] |

## 관련 개념
- [[관련 페이지 1]]
- [[관련 페이지 2]]

## 출처
- [[sources_페이지명]] — {한 줄 설명}
```

**저장 경로 결정**:
- 개념/기술 → `wiki/concepts/`
- 인증/기준/표준 → `wiki/standards/`
- 설비/장비 → `wiki/equipment/`
- 사람 → `wiki/people/`
- 회사/기관 → `wiki/companies/`
- 지표 → `wiki/metrics/`
- 프로젝트 → `wiki/projects/`

> **규칙**: 모든 사실 진술에 `[[sources_페이지명]]` 인용 필수 (INV-5).

---

#### 3-3. 모순 처리

Ingest 중 기존 페이지와 새 자료 간 사실 충돌 발견 시:

1. **임의 판정 금지.** 어느 쪽이 옳은지 결정하지 않는다.
2. 해당 개념 페이지 하단에 추가:

```markdown
## ⚠️ 검토 필요 (모순)

**쟁점**: {충돌하는 항목 설명}

- [[소스A]]: {주장A} (p.N)
- [[소스B]]: {주장B} (p.N)

**상태**: 미해결. 사용자 검토 필요.
**기록일**: {YYYY-MM-DD}
```

3. 사용자 보고 시 모순 항목을 별도로 강조.

---

#### 3-4. 상호 참조 갱신

- 새 페이지에서 기존 페이지로 가는 `[[링크]]` 확인 및 추가.
- 기존 페이지에서 새 페이지로 오는 링크가 누락된 경우 해당 페이지에 추가.
- 모든 내부 참조는 `[[wikilink]]` 형식 사용 (일반 마크다운 링크 금지).

---

### STEP 4 — INDEX.md 갱신 (INV-3 필수)

페이지 생성/업데이트마다 `INDEX.md` 를 즉시 갱신한다.

**항목 형식 (120자 이내, 1줄)**:
```
- [[페이지명]] — 1줄 요약 (sources: N)
```

- 신규 페이지 → 해당 카테고리 섹션에 추가
- 페이지 수 갱신: `마지막 갱신: {날짜} | 페이지 수: {N}`
- `<!-- 섹션명 페이지 -->` 주석만 있던 섹션에 첫 항목 추가 시 주석 제거

---

### STEP 5 — log.md 추가 (INV-4 필수)

오퍼레이션 종료 직전, `log.md` 에 append:

```markdown
## [{YYYY-MM-DD}] ingest | {파일명}
- 출처: {저자/발행처} ({URL — 해당 시})
- 신규 (sources/): [[{sources_페이지명}]]
- 신규 (concepts/): [[{개념1}]], [[{개념2}]]  ← 해당 시
- 업데이트 (concepts/): [[{기존페이지}]] — {무엇을 보강했는지}  ← 해당 시
- 모순: {없음 | [[페이지명]] — 쟁점 요약}
- 주의: {특이사항 — 해당 시}
- INDEX 갱신: ✓ (페이지 수 {이전} → {이후})
```

> `log.md` 기존 항목은 수정 금지. append only.

---

### STEP 6 — 사용자 보고

최종 요약을 다음 형식으로 보고한다:

```
## Ingest 완료: {파일명}

### 신규 생성
- sources/: [[{페이지명}]]
- concepts/: [[{페이지명}]], [[{페이지명}]]
- (해당 시) people/standards/equipment/: [[{페이지명}]]

### 업데이트
- [[{기존 페이지명}]] — {보강 내용 한 줄}

### 검토 필요
- ⚠️ 모순: [[{페이지명}]] — {쟁점 요약}  ← 해당 시
- ⚠️ 데이터 공백: {주제} — {공백 내용}  ← 해당 시
- needs-review: [[{페이지명}]]  ← 해당 시

### INDEX 갱신
페이지 수: {이전} → {이후}
```

## 다음 파일 처리

미처리 파일이 여럿인 경우, 한 파일 완료 후:

> "다음 파일 [{파일명}] 을 처리할까요? (y / 잠깐 멈춰 / 나중에)"

사용자 확인 후 STEP 1부터 반복.

## 자주 있는 예외 상황

| 상황 | 처리 |
|------|------|
| 웹 클리퍼 부분 캡처 | `status: needs-review`, `⚠️ 데이터 공백` 섹션 추가 |
| 동일 개념이 여러 alias로 검색됨 | 가장 대표적인 한국어명으로 통일, 나머지는 aliases 등록 |
| LLM 일반 지식으로 보충 불가피 | `_(일반 상식 보충)_` 표기 + `confidence: low` |
| raw/ 파일이 이미지·바이너리 | 내용 파악 불가 사실 사용자에게 알림 후 건너뜀 |
| 소스 출처 불분명 | `confidence: low`, 알려진 정보만 기록, 추측 금지 |
| 카테고리 신설 필요 | 사용자 승인 없이 신규 카테고리 폴더/섹션 생성 금지 |

## 금지 사항 (요약)

- ❌ `raw/` 파일 수정·삭제·이동
- ❌ `sources/` 페이지에 해석·종합·비교 작성
- ❌ 개념 페이지에 sources/ 인용 없는 사실 진술
- ❌ 기존 페이지 내용 통째 덮어쓰기
- ❌ 모순 발견 시 임의 판정 또는 한쪽 소스만 인용
- ❌ INDEX.md 항목 120자 초과
- ❌ log.md 기존 항목 수정
- ❌ 내부 링크를 일반 마크다운 `[text](url)` 로 작성
- ❌ 오퍼레이션 종료 시 INDEX.md / log.md 미갱신
